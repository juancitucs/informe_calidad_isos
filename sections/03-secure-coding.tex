\section{Secure Coding}
\label{sec:secure-coding}

El control 8.28 de ISO/IEC 27001:2022 establece que la organización debe establecer principios de codificación segura y aplicarlos al desarrollo de software para reducir la introducción de vulnerabilidades durante la implementación \cite{iso27001-wiki}.

\subsection{Filosofía}

Muchos ataques exitosos no son sofisticados. Son errores simples como SQL Injection, XSS, Path Traversal, Command Injection o CSRF. Todos nacen del código. Por eso ISO 27001 crea un control exclusivo para la programación segura.

El objetivo de 8.28 no es que cada programador sea un experto en seguridad, sino que existan \textbf{reglas claras, documentadas y aplicables} que reduzcan la superficie de ataque desde la fase de implementación.

\subsection{Principios Fundamentales de Codificación Segura}

La organización debe establecer principios de codificación segura aprobados. A continuación se describen los principios más relevantes:

\subsubsection{1. Nunca confiar en la entrada}

La entrada del usuario siempre debe considerarse no confiable hasta que sea validada. Esto aplica a:

\begin{itemize}
    \item Parámetros de URL y query strings.
    \item Cuerpos de peticiones HTTP (JSON, form-data).
    \item Headers y cookies.
    \item Archivos subidos por el usuario.
    \item Datos de bases de datos externas.
\end{itemize}

\textbf{Ejemplo genérico de código inseguro:}

\begin{lstlisting}[language=Java, caption={Código inseguro: sin validación de entrada}]
// INSEGURO - No validación de entrada
app.get('/api/items/:id', (req, res) => {
  const item = db.query('SELECT * FROM items WHERE id = '
    + req.params.id);
  res.json(item);
});
\end{lstlisting}

\textbf{Solución genérica:}

\begin{lstlisting}[language=Java, caption={Código seguro: con validación de entrada}]
// SEGURO - Con validación y consulta parametrizada
app.get('/api/items/:id', (req, res) => {
  if (!isValidId(req.params.id)) {
    return res.status(400).json({ error: 'ID invalido' });
  }
  const item = db.query(
    'SELECT * FROM items WHERE id = ?', [req.params.id]
  );
  res.json(item);
});
\end{lstlisting}

\subsubsection{2. Validar siempre}

Validar significa comprobar que el dato cumple lo esperado antes de procesarlo. Se deben validar: tipo, longitud, formato, rango y conjunto permitido de valores.

\begin{lstlisting}[language=Java, caption={Ejemplo genérico de validación de entrada}]
const { body, validationResult } = require('express-validator');

app.post('/api/register', [
  body('email').isEmail().normalizeEmail(),
  body('password').isLength({ min: 8 })
    .matches(/^(?=.*[A-Z])/),
  body('name').trim().escape()
    .isLength({ max: 100 }),
], (req, res) => {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    return res.status(400).json({
      errors: errors.array()
    });
  }
  // Procesar registro...
});
\end{lstlisting}

\subsubsection{3. Sanitizar cuando corresponda}

Sanitizar significa preparar los datos para su uso seguro. Un input como \texttt{<script>alert(1)</script>} puede producir Stored XSS si se almacena sin control.

En frameworks modernos como React, la salida se escapa automáticamente. Sin embargo, hay funciones que deben evitarse:

\begin{lstlisting}[language=Java, caption={Uso inseguro de dangerouslySetInnerHTML}]
// INSEGURO - Nunca usar sin sanitizar
const UserBio = ({ bio }) => {
  return <div
    dangerouslySetInnerHTML={{__html: bio}}
  />;
};

// SEGURO - React escapa automáticamente
const UserBio = ({ bio }) => {
  return <div>{bio}</div>;
};
\end{lstlisting}

\subsubsection{4. Principio de mínimo privilegio}

El software debe usar únicamente los permisos que necesita para operar. Esto aplica a:

\begin{itemize}
    \item \textbf{Bases de datos:} El usuario de la aplicación debe tener permisos solo sobre la base de datos de la aplicación, no sobre administración del servidor.
    \item \textbf{Almacenamiento:} Las credenciales de acceso a almacenamiento deben limitarse al bucket o directorio necesario.
    \item \textbf{Contenedores:} Los contenedores deben ejecutarse con usuarios no root y exponer solo los puertos estrictamente necesarios.
    \item \textbf{Archivos de configuración:} Los permisos de archivos \texttt{.env} deben ser restringidos al proceso de la aplicación.
\end{itemize}

\subsubsection{5. Gestión de secretos}

Nunca almacenar secretos en código fuente ni en repositorios:

\begin{itemize}
    \item Usar archivos de variables de entorno (ej: \texttt{.env}) excluidos del control de versiones.
    \item Crear archivos plantilla versionados (ej: \texttt{.env.example}) como referencia.
    \item Cargar variables de entorno con librerías dedicadas (ej: \texttt{dotenv}).
    \item Nunca: \texttt{const PW = "admin123"} en código.
    \item En producción: usar gestores de secretos como HashiCorp Vault, AWS Secrets Manager o Azure Key Vault.
\end{itemize}

\subsubsection{6. Manejo seguro de errores}

Los mensajes de error no deben exponer información sensible del sistema:

\begin{lstlisting}[language=Java, caption={Manejo seguro de errores}]
// INSEGURO - Expone detalles del error
app.use((err, req, res, next) => {
  res.status(500).json({
    error: err.message,
    stack: err.stack  // NUNCA exponer
  });
});

// SEGURO - Mensaje genérico, log interno
app.use((err, req, res, next) => {
  console.error(`[${new Date()}] ${err.message}`);
  res.status(500).json({
    error: 'Error interno del servidor'
  });
});
\end{lstlisting}

\subsubsection{7. Dependencias seguras}

Gran parte de los ataques actuales no vienen del código propio, sino de dependencias externas. La organización debe establecer procesos para:

\begin{itemize}
    \item Escanear dependencias periódicamente para vulnerabilidades conocidas.
    \item Actualizar paquetes con vulnerabilidades identificadas.
    \item Eliminar dependencias abandonadas o innecesarias.
    \item Usar herramientas automáticas de escaneo (ej: \texttt{npm audit}, \texttt{pip-audit}, \texttt{safety}).
    \item Definir una política de actualización de dependencias.
\end{itemize}

\subsection{Qué busca un auditor en 8.28}

Un auditor de ISO/IEC 27001 busca evidencias de:

\begin{itemize}
    \item Documentación de principios de codificación segura aprobados.
    \item Guías de codificación segura disponibles para los desarrolladores.
    \item Plugins de seguridad en herramientas de desarrollo (linters, SAST).
    \item Revisiones de código con checklist de seguridad.
    \item Ausencia de secretos hardcodeados en el código fuente.
    \item Uso de librerías y dependencias actualizadas.
    \item Registros de escaneo de dependencias.
\end{itemize}
